Plan de développement CRM

Ce plan remplace l'ancienne feuille de route basée sur le squelette. Le vrai point de départ est le CRM déjà déployé sur le VPS, et le travail consiste désormais à préserver, migrer, aligner et étendre ce produit jusqu'à ce qu'il corresponde à la cible finale Aqua Plus.

Produit en production Un CRM React + Vite + Supabase existe déjà sur le VPS à /var/www/crm.
Stack confirmé React reste. Supabase reste. Le squelette Vue sera remplacé par le code React du VPS.
Tâche principale Importer le code React du VPS dans le dépôt, puis aligner et étendre.

Décisions d'architecture (6 avril 2026)

  • React reste. Le code React du VPS est importé dans apps/web. Le squelette Vue est supprimé.
  • Supabase reste. Auth, Storage, PostgREST et Postgres restent en place comme couche backend principale.
  • Contacts = prospects catégorisés. Pas d'entité séparée. Un champ category est ajouté : client, fournisseur, employé.
  • Express API réduit. apps/api est réservé aux tâches côté serveur : webhooks, jobs, automatisations, intégrations.

Base de planification

La cible finale est définie dans docs/product/final-crm-spec.md. La référence d'implémentation est le CRM React déployé sur le VPS.

  • Importer le code React tel quel depuis le VPS.
  • Préserver le comportement utile existant avant d'élargir le périmètre.
  • Supabase gère le CRUD — l'API Express ne gère que les tâches côté serveur.
  • Maintenir les éléments exclus hors périmètre : e-commerce, inventaire, fidélité, codes promo et outils de recommandation.

Stratégie de livraison

1. Capturer la référence produit (essentiellement terminé).
2. Importer le code React du VPS dans le dépôt sans régression.
3. Aligner les modules existants sur la terminologie et le flux de travail Aqua Plus.
4. Ajouter la catégorisation des contacts et les modules opérationnels manquants.
5. Durcir, basculer et exploiter le système depuis ce dépôt.

Phases

Phase 1 — essentiellement terminée

Capture de la référence produit et alignement du dépôt

L'audit du 3 avril et les décisions d'architecture du 6 avril couvrent cette phase. Le stack est confirmé (React + Supabase), la stratégie d'adoption est définie (import direct).

Phase 2

Import du code React dans le dépôt sans régression

Copier le code React de /var/www/crm dans apps/web, supprimer le squelette Vue, configurer les variables d'environnement Supabase, vérifier que le build fonctionne et que tous les modules existants sont opérationnels. Configurer apps/api comme service Express léger pour les futures tâches côté serveur.

Phase 3

Alignement sur le flux de travail Aqua Plus

Remplacer le langage CRM générique actuel par le vrai modèle opérationnel Aqua Plus : étapes, types de services, catégorisation des contacts, profils de rôles et visibilité de la source des prospects.

Phase 4

Catégorisation des contacts et dossier client

Ajouter un champ de catégorie aux prospects (client, fournisseur, employé), diviser la page contacts par catégorie, et construire un dossier client unifié couvrant l'historique commercial complet.

Phase 5

Intégration du site web, automatisations et notifications

Ajouter la capture de prospects depuis le site web, l'attribution de la source, les confirmations automatiques, les alertes internes de prospects, les notifications de statut et la logique de relance basée sur l'étape ou l'inactivité.

Phase 6

Prise de rendez-vous publique et intégration de la planification

Ajouter la disponibilité des vendeurs, la prise de rendez-vous publique, la validation des rendez-vous, la synchronisation du calendrier interne, la liaison des enregistrements CRM et les notifications de rendez-vous au personnel.

Phase 7

Facturation et opérations de paiement

Étendre le moteur de devis existant en un flux de facturation complet avec génération de factures, conversion devis-en-facture, obligations de paiement, enregistrements de paiements et visibilité des dates d'échéance depuis le dossier client.

Phase 8

Durcissement et bascule vers la production

Ajouter la vérification automatisée, les contrôles de migration, la planification du retour arrière et la documentation de déploiement pour que le CRM en production puisse être géré depuis ce dépôt avec un chemin de livraison crédible.

Principaux risques

  • L'import du code React du VPS peut nécessiter des ajustements de chemins, de variables d'environnement ou de dépendances non documentées.
  • Les devis sont l'un des domaines à plus haut risque car ils combinent calculs, modèles, PDFs, signatures, paramètres et stockage.
  • Modifier les étapes et les rôles trop tôt peut créer des perturbations opérationnelles pendant la migration.
  • Les automatisations ne doivent pas être livrées tant que la propriété des événements et le comportement d'audit ne sont pas clairs.
  • L'API Express doit rester légère pour ne pas dupliquer les responsabilités de Supabase.

Critères de sortie

  • Le dépôt est basé sur le vrai CRM de production plutôt que sur le squelette.
  • Les modules implémentés actuels sont préservés sans régression.
  • La terminologie, les étapes, les services et les profils de rôles d'Aqua Plus sont reflétés dans le produit.
  • Les prospects issus du site web entrent automatiquement dans les flux CRM.
  • Les notifications et les automatisations de relance sont opérationnelles et auditables.
  • La prise de rendez-vous publique et la planification interne sont synchronisées.
  • Les factures et le suivi des paiements sont visibles depuis le dossier client.
