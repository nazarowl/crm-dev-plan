CRM Development Plan

This plan replaces the old scaffold-first roadmap. The real starting point is the deployed CRM already running on the VPS, and the job is now to preserve, migrate, align, and extend that product until it matches the final Aqua Plus target.

Live Product React + Vite + Supabase CRM already exists on the VPS at /var/www/crm.
Local Repo The current apps/web and apps/api remain scaffold-level only.
Main Task Stop planning like a blank-slate build. Plan from the real shipped system.

Planning Baseline

The final target is defined in docs/product/final-crm-spec.md. The implementation baseline is the deployed CRM inspected on the VPS, not the scaffold in this repository.

  • Use the deployed CRM as the functional reference.
  • Preserve useful working behavior before broadening scope.
  • Keep excluded items out of scope: e-commerce, inventory, loyalty, promo codes, and recommendation tooling.

Delivery Strategy

1. Capture and stabilize the real existing product baseline.
2. Bring that baseline into this repository without regression.
3. Align the existing modules to Aqua Plus terminology and workflow.
4. Add the missing operational modules.
5. Harden, cut over, and run the system from this repository.

Phases

Phase 1

Product Baseline Capture and Repository Alignment

Treat the VPS CRM as the real source product. Document its modules, data model, Supabase dependencies, and deployment constraints, then define how that product becomes repo-managed code instead of a separate server-side codebase.

Phase 2

Core Product Migration Without Regression

Import or reproduce the deployed CRM inside this repo and preserve the current working modules: auth, roles, permissions, leads, activities, pipeline, calendar, quotes, templates, settings, attachments, and admin behavior.

Phase 3

Aqua Plus Workflow Alignment

Replace the current generic CRM language with the real Aqua Plus operating model: stages, service types, contact categorization, role profiles, and lead source visibility.

Phase 4

Customer Record and Dossier Consolidation

Resolve the current lead-versus-contact ambiguity and build a unified customer record that can hold long-term commercial history across opportunities, activities, quotes, bookings, invoices, and payments.

Phase 5

Website Intake, Automations, and Notifications

Add website lead capture, source attribution, automatic confirmations, internal lead alerts, status notifications, and stage or inactivity-based follow-up logic.

Phase 6

Public Booking and Scheduling Integration

Add seller availability, public booking, booking validation, internal calendar sync, CRM record linking, and staff booking notifications.

Phase 7

Invoicing and Payment Operations

Extend the existing quote engine into a complete billing workflow with invoice generation, quote-to-invoice conversion, payment obligations, payment records, and due-date visibility from the customer dossier.

Phase 8

Release Hardening and Cutover

Add automated verification, migration checks, rollback planning, and deployment documentation so the production CRM can be managed from this repository with a credible release path.

Main Risks

  • The old roadmap overstates greenfield work and understates migration work.
  • Quotes are one of the highest-risk areas because they combine calculations, templates, PDFs, signatures, settings, and storage.
  • The customer/contact model needs an explicit decision before billing and booking expand further.
  • Changing stages and roles too early can create operational churn during migration.
  • Automations should not ship until event ownership and audit behavior are clear.

Exit Criteria

  • The repository is based on the real production CRM rather than the scaffold.
  • Current implemented modules are preserved without regression.
  • Aqua Plus terminology, stages, services, and role profiles are reflected in the product.
  • Website-originated leads enter CRM workflows automatically.
  • Notifications and follow-up automations are operational and auditable.
  • Public booking and internal scheduling are synchronized.
  • Invoices and payment tracking are visible from the client dossier.
